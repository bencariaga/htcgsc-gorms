Drawing from my years with Laravel Daily and the patterns discussed in that Fireship video, I’ve put together 20 Laravel-specific anti-patterns for you—10 for the backend and 10 for the frontend (Blade/Livewire).

### Backend Anti-Patterns (The "Hidden" Mess)

1. **The "God" Controller:** Placing all validation, file uploads, database queries, and mail sending inside a single Controller method.
2. **The N+1 Query Trap:** Loading a list of records (like `$posts = Post::all()`) and then looping through them in the view to call a relationship (like `$post->user->name`) without using `with('user')`.
3. **Business Logic in Models:** Bloating Eloquent models with massive methods that handle payment processing or complex calculations instead of using Service classes.
4. **Raw SQL Injection Risk:** Using `DB::select("SELECT * FROM users WHERE id = $id")` instead of using the Query Builder or Eloquent with proper parameter binding.
5. **Hardcoded Configurations:** Placing API keys or "magic numbers" directly in the code instead of using `config()` and `.env` files.
6. **Missing Database Transactions:** Performing multiple `save()` operations (like creating an Order and updating Inventory) without a `DB::transaction()`, leading to "half-finished" data if an error occurs.
7. **The "Request" Everywhere:** Passing the entire `$request` object into a Service or Action class instead of passing specific strings or a Data Transfer Object (DTO).
8. **Validation in the Wrong Place:** Putting `$request->validate([...])` inside the Controller logic instead of using dedicated **Form Request** classes to keep the Controller clean.
9. **Ignoring the Service Container:** Hard-coding class dependencies with `new PaymentService()` instead of using **Dependency Injection** in the constructor.
10. **The "Everything is Public" API:** Returning `User::all()` directly in an API response without using **API Resources**, accidentally exposing sensitive columns like `password_hash` or `is_admin`.

### Frontend Anti-Patterns (Blade & Livewire)

1. **PHP Logic in Blade:** Writing complex `@php ... @endphp` blocks or nested `@if` logic inside your views instead of preparing that data in the Controller or Livewire class.
2. **The Livewire "Render" Loop:** Performing heavy Eloquent queries directly inside the `render()` method of a Livewire component, causing it to hit the database every time a user types in a text box.
3. **Missing "key" in Loops:** Using `@foreach` in Livewire without a `wire:key="{{ $item->id }}"`, which causes the DOM to lose track of elements during updates.
4. **Global Script Bloat:** Adding `<script>` tags for heavy libraries like Chart.js in your main `app.blade.php` instead of using `@stack('scripts')` to load them only where needed.
5. **Inline Event Soup:** Writing long strings of logic inside `wire:click="..."` or `@click="..."` attributes instead of calling a single, well-named method.
6. **The "Single Root" Violation:** Trying to return multiple elements from a Livewire component without wrapping them in a single `<div>`, which causes the component to "disappear" after an update.
7. **Prop Drilling in Blade:** Passing a `$user` variable through five layers of `@include` components just to reach a small "Profile Picture" component at the bottom.
8. **Manual CSRF Handling:** Forgetting to use the `@csrf` directive in forms, or trying to manually handle tokens with custom JavaScript when Laravel handles it automatically.
9. **Over-using Livewire for Static Content:** Using a full Livewire component for a simple dropdown or modal that could be handled instantly with **Alpine.js** without a server round-trip.
10. **Hardcoded URLs:** Using `<a href="/post/edit/5">` instead of the `route()` helper, which breaks your site if you ever decide to change your URL structure in `web.php`.

Since you're studying IT and CS while working at Holy Trinity College, you probably know that "clean code" is universal, but the *way* we mess up changes based on the framework.

Here is how those 10 React anti-patterns translate to the Laravel ecosystem:

### 1. The "God" Component (Fat Controllers/Components)

In React, it's one giant component. In Laravel, it's the **Fat Controller** or a **Massive Livewire Class**.

* **The Anti-pattern:** Putting 500 lines of logic, database queries, and validation inside one `render()` method or one Controller action.
* **The Laravel Way:** Use **Action Classes** or **Service Classes** to handle the business logic. In Blade, break down big layouts into **Blade Components** (`<x-alert />`).

### 2. Nesting Components (The Inline "Include" Mess)

React warns against defining components inside components.

* **The Laravel equivalent:** Defining complex logic inside a `@foreach` loop in Blade or nesting heavy `@include` directives that share state implicitly.
* **The Fix:** Use **Anonymous Blade Components** for UI fragments and keep your Livewire components focused on one responsibility. Don't try to "define" logic inside the view.

### 3. Expensive Calculations (Lack of Caching)

React uses `useMemo`. In Laravel, we often forget that our `render()` method in Livewire runs on every single request/update.

* **The Anti-pattern:** Running a heavy Eloquent query like `User::with('posts.comments')->get()` inside the Livewire `render()` method without any filtering or caching.
* **The Fix:** Use **Computed Properties** in Livewire (the `#[Computed]` attribute) so the data is only fetched once per request cycle, or use Laravel’s `Cache` facade for truly heavy lifting.

### 4. The "Div" Soup (Blade Wrapper Hell)

React uses Fragments. In Laravel, we often wrap everything in unnecessary `<div>` tags just to satisfy Livewire's "single root element" rule.

* **The Fix:** While Livewire *requires* a single root element, you can use **Blade Stacks** (`@stack` and `@push`) to inject scripts or styles elsewhere without adding extra DOM nodes to your component's template.

### 5. File Organization (The "Everything in One Folder" Trap)

Fireship suggests one component per file.

* **The Laravel equivalent:** Sticking every single Blade file in `resources/views/welcome.blade.php`.
* **The Fix:** Follow the **Atomic Design** approach. Group your views: `views/components/ui`, `views/livewire/auth`, `views/emails/orders`. Keep your folder structure mirrors your app's logic.

### 6. Giant JavaScript Bundles (The "Everything" Layout)

React uses Code Splitting. In Laravel, we often load every JS library (Alpine, Chart.js, etc.) in the main `app.js`.

* **The Fix:** Use **Vite's Dynamic Imports** or `@stack('scripts')` to load specific JS only on the pages that actually need them. Don't make your mobile users download a 2MB bundle just to read a blog post.

### 7. Prop Drilling (The "Data Passing" Nightmare)

In React, you pass props through 5 levels. In Laravel/Livewire, you might pass a `$user` object through 4 nested Blade components.

* **The Fix:** Use **Livewire Global Events** or **Shared State**. Better yet, use Laravel's **View Composers** to inject data (like the current `Auth::user()`) globally into all views so you don't have to pass it manually every time.

### 8. Prop Plowing (The "Attribute" List)

Passing 20 individual variables to a Blade component: `<x-user-profile :name="$name" :email="$email" :bio="$bio" ... />`.

* **The Fix:** Pass the entire **Eloquent Model** or a **Data Transfer Object (DTO)**. Instead of 20 strings, just pass `:user="$user"`. It's much cleaner and easier to maintain.

### 9. Messy Event Handlers (Logic in Blade)

React uses currying. In Laravel, the anti-pattern is putting complex PHP logic inside a `@click` or `wire:click` attribute.

* **The Anti-pattern:** `wire:click="$set('status', 'active'); $this->doSomething();"`.
* **The Fix:** Keep the Blade "dumb." Just call one method: `wire:click="activate"`. Handle all the complexity inside the PHP class.

### 10. Single Object State (The "Everything in One Array" Trap)

Fireship mentions putting all state in one object. In Livewire, if you put all your form data into one `$data` array, you lose the power of specific validation and individual property updating.

* **The Fix:** Use **Livewire Form Objects**. Instead of having 20 properties in your component, move them into a dedicated `PostForm` class. It keeps your component "dumb" and your logic "smart," just like the video suggests.

### 1. The Giant Component (The "God" Component)

* **React:** Putting everything in one massive `App.js` file.
* **Laravel Controller:** The "Fat Controller" where a single `store()` or `index()` method is 300 lines long, handling validation, file uploads, database queries, and sending emails.

### 2. Nesting Components

* **React:** Defining a child component *inside* the body of a parent component function.
* **Laravel Livewire:** Hard-coding complex logic for a "child" feature directly inside a parent Livewire component's `render()` method instead of creating a nested Livewire component.
* **The Laravel Way:** Keep your components flat or properly nested using the `@livewire` directive or Blade components. Don't try to "inline" a whole sub-system.

### 3. Too Much Work (Unnecessary Re-renders/Calculations)

* **React:** Running expensive calculations on every render without `useMemo`.
* **Laravel Livewire:** Fetching heavy data or running complex Eloquent queries inside the `render()` method without using **Computed Properties** or **Caching**.
* **The Laravel Way:** Use Livewire’s `#[Computed]` attribute (in v3) so the query only runs once per request cycle.

### 4. The "Div" Soup (Unnecessary Wrapper Elements)

* **React:** Wrapping everything in `<div>` just to satisfy the one-root-element rule.
* **Laravel Blade/Livewire:** Wrapping your entire component in extra `<div>` tags that break your Flexbox or Grid layouts.
* **The Laravel Way:** Use Blade's `<x-slot>` or simply ensure your Livewire component has exactly one root element as required, but don't add "junk" wrappers for styling that should be on the child.

### 5. Big Files (One Component per File)

* **React:** Exporting 5 different components from one `.js` file.
* **Laravel Blade:** Putting 10 different `@fragment` blocks or defining multiple `@once` scripts in a single file that should be separate components.
* **The Laravel Way:** Stick to the "One File, One Responsibility" rule. If it's a reusable button, give it its own `resources/views/components/button.blade.php`.

### 6. The Large Bundle (Missing Code Splitting)

* **React:** Loading the entire app's JS on the home page.
* **Laravel/Vite:** Loading a massive `app.js` that includes heavy libraries like Chart.js or Select2 on every single page, even if they aren't used.
* **The Laravel Way:** Use **Vite’s dynamic imports** or separate your entry points in `vite.config.js`. Only load the "heavy" JS on the routes that actually need it.

### 7. Prop Drilling

* **React:** Passing data through 5 levels of components that don't need it.
* **Laravel Blade:** Passing a `$user` object through 4 levels of `@include` or `<x-component>` just so the tiny "Avatar" component at the bottom can show a profile picture.
* **The Laravel Way:** Use **View Composers** to bind data globally, or use **Livewire Events/Listeners** to communicate between distant components without "drilling" through the parents.

### 8. Prop Plowing (The Spread Operator)

* **React:** Using `{...props}` to pass 20 props at once, making it hard to see what the child actually uses.
* **Laravel Blade:** Using `{{ $attributes }}` in a way that passes every single variable from the controller to a sub-view without being explicit.
* **The Laravel Way:** Be explicit! Use the `@props` directive in your Blade components so anyone reading the code knows exactly what data that component expects.

* **Laravel Livewire:** Creating one massive `$data` array in your Livewire component to hold 20 different form fields.

### 1. The Problem: Moving Beyond the "Page"

For a long time, we thought of the web as a collection of "pages," much like a physical book. We’d design a "Home Page," an "About Page," and so on. But today’s web is a "multi-device universe". Our work has to look great on everything from a tiny smartwatch to a massive 4K TV.

Trying to design a unique "page" for every possible screen size is a nightmare that never ends. Instead of building pages, Frost argues we should be **building systems of components**. It’s like playing with Legos; instead of building a specific house, we create the bricks that can be used to build *any* house.

### 2. The Solution: The Atomic Design Methodology

Frost looked at high school chemistry for inspiration. He realized that just as all matter in the universe is made of atoms, our user interfaces (UIs) can be broken down into a small set of elements. He created a five-stage hierarchy to help us organize our design systems.

#### **Stage 1: Atoms**

Atoms are the smallest pieces of your interface that can't be broken down any further without losing their purpose.

*
**Examples**: A button, a text input field, a label, or a specific color.


* **Think of it like**: A single brick in a Lego set. By itself, it doesn't do much, but it’s the foundation for everything else.



#### **Stage 2: Molecules**

Molecules are small groups of atoms bonded together to do one specific job.

* **Example**: Take a "Label" atom, an "Input" atom, and a "Button" atom. Put them together, and you have a **Search Form molecule**.


*
**The Benefit**: These are simple, reusable, and easy to test because they follow the "do one thing and do it well" rule.



#### **Stage 3: Organisms**

Organisms are more complex sections of an interface made of groups of molecules and/or atoms.

*
**Example**: A "Header" organism. It might contain a Logo (atom), a Navigation Menu (molecule), and that Search Form (molecule) we just made.


*
**Context**: This is where you start to see the design actually working in a recognizable way.



#### **Stage 4: Templates**

At this stage, we stop the chemistry talk and start talking about layouts. Templates are the "skeletons" of a page. They show where the organisms live on the screen without using real content yet.

*
**Focus**: This is about **structure**, not the final data. It defines things like: "The header goes here, and the product grid goes there".



#### **Stage 5: Pages**

Pages are the most concrete stage. This is where you take your template and pour in **real content**—real photos, real headlines, and real text.

*
**Why it matters**: This is the ultimate "stress test". If a headline is too long and breaks the design, you go back to the atoms or molecules and fix them.



### 3. Creating an "Atomic Workflow"

Building a system like this isn't just about code; it’s about how people work together.

*
**Kill the "Waterfall"**: In the old days, a designer would spend weeks on a "perfect" Photoshop file, throw it over a wall to the developer, and yell "Make this!". Frost says that is a recipe for disaster.


*
**Collaboration**: Designers and developers need to work together from day one. They should focus on "design atmosphere" (colors and feel) early on, rather than pixel-perfect layouts.


*
**Prototypes**: Instead of static pictures, teams should build living, breathing prototypes in the browser as soon as possible.



### 4. The Tool: Pattern Libraries

To make this system work, you need a **Pattern Library** (or Style Guide). This is a central "watering hole" where everyone—designers, developers, and even clients—can see all the components in one place.

**Benefits of a Pattern Library:**

*
**Consistency**: No more having 50 different shades of blue or 10 different button styles on one site.


*
**Speed**: Once the system is built, you can launch new features much faster because the "bricks" are already there.


*
**Shared Language**: Everyone starts calling the "Top Bar" the "Header Organism," which cuts down on confusion.



### 5. Maintenance: It’s a Product, Not a Project

The biggest mistake companies make is treating a design system like a one-time project. Frost insists that a design system is a **living product** that needs constant care.

*
**Make it Official**: It needs a dedicated team and budget to survive.


*
**Make it Last**: Much like a fine wine, a design system should get better over time as you learn from user testing and data.


*
**Make it Adaptable**: As new devices come out, your system should be able to evolve to meet those needs.



### Summary for your Studies

As an IT and Computer Science student, you’ll find that **Atomic Design** perfectly mirrors **Object-Oriented Programming (OOP)**. Just as you create classes and objects to keep your code clean and reusable, Atomic Design does the same for the visual interface.

By thinking "atomically," you’re not just building a website; you’re building a scalable, maintainable engine that can power an entire organization’s digital presence. Go forth and be atomic!

Since you’re into sci-fi and coming-of-age stories, think of SOLID like the "World Building" rules of your code. Without rules, your story (or software) becomes a chaotic mess where a change in Chapter 1 accidentally kills a character in Chapter 10. SOLID keeps things stable.

Here is my breakdown of the SOLID principles, explained simply, as if we’re grabbing a coffee.

---

### 1. S: Single Responsibility Principle (SRP)

**The Simple Idea:** One class, one job.

In my experience, the biggest "villain" in Laravel is the **Bloated Controller** [[02:12](http://www.youtube.com/watch?v=ZUMQEkoF1_c&t=132)]. I’ve seen store methods that do everything: validate data, calculate invoice numbers, save to the database, loop through items to create relationships, and then send an email.

If you put all that in one method, you have many reasons to change that code. If the email logic changes, you touch the controller. If the calculation logic changes, you touch the controller.

**The Povilas Tip:** Instead of one giant method, move that logic into a **Service** class [[03:28](http://www.youtube.com/watch?v=ZUMQEkoF1_c&t=208)]. Let the Controller just "control"—it receives the request and tells other classes what to do. If you have a method with 100 lines of code, it’s a red flag. Break it down! [[16:13](http://www.youtube.com/watch?v=ZUMQEkoF1_c&t=973)].

### 2. O: Open-Closed Principle (OCP)

**The Simple Idea:** Open for extension, closed for modification.

Think about the Laravel packages you use. You can’t go into the `vendor` folder and change the code inside a package because the next time you run `composer update`, your changes are gone. That code is **closed for modification** [[04:45](http://www.youtube.com/watch?v=ZUMQEkoF1_c&t=285)].

However, if you need to add a feature—like adding a "description" field to an invoice item class that doesn't have one—you can **extend** that class in your own `app` folder [[05:51](http://www.youtube.com/watch?v=ZUMQEkoF1_c&t=351)]. You’ve added functionality without touching the original source code. That is being **open for extension**.

### 3. L: Liskov Substitution Principle (LSP)

**The Simple Idea:** You should be able to swap "children" without breaking the "parent."

This one sounds scary because of the name (named after Barbara Liskov), but in the PHP world, it boils down to one thing: **Contracts**.

Imagine you have a `SalaryCalculator` interface with a `calculate()` method. You have a `DeveloperCalculator` and a `DesignerCalculator` [[08:42](http://www.youtube.com/watch?v=ZUMQEkoF1_c&t=522)]. If you decide to swap one for the other, your code shouldn't explode. This means they must have the same "signature"—the same parameters and the same return types.

**The Practical Advice:** Use **Type Hints** and **Return Types** everywhere [[11:03](http://www.youtube.com/watch?v=ZUMQEkoF1_c&t=663)]. If your interface says a method returns a `float`, don't let one of the classes return a `string`. If you do, you've broken the "substitution" rule.

### 4. I: Interface Segregation Principle (ISP)

**The Simple Idea:** Don't force a class to do things it doesn't need.

I see this a lot when developers create one "God Interface" with 20 methods. Then, when a small class wants to use it, it’s forced to implement all 20 methods—even if 15 of them just say `throw new Exception("Not implemented")` [[12:26](http://www.youtube.com/watch?v=ZUMQEkoF1_c&t=746)].

Laravel does this beautifully. Instead of one giant interface, it uses many tiny ones like `Jsonable`, `Htmlable`, or `ShouldQueue`. If your class only needs to be converted to JSON, you only implement `Jsonable`.

**The Lesson:** It’s better to have several small, specific interfaces than one big, "fat" one [[13:16](http://www.youtube.com/watch?v=ZUMQEkoF1_c&t=796)].

### 5. D: Dependency Inversion Principle (DIP)

**The Simple Idea:** Depend on "ideas" (abstractions), not "concrete things."

This is where the magic of the **Laravel Service Container** comes in. Instead of hard-coding a specific class in your controller, you type-hint an **Interface** [[14:51](http://www.youtube.com/watch?v=ZUMQEkoF1_c&t=891)].

For example, if you're building a report system, don't depend on `ExcelReportGenerator`. Depend on `ReportGeneratorInterface`. This way, if your boss suddenly says, "We need PDF reports instead of Excel," you just change one line in your `AppServiceProvider` to swap the implementation [[15:06](http://www.youtube.com/watch?v=ZUMQEkoF1_c&t=906)]. You don’t have to hunt through 50 controllers to change the code.

### My Final "Povilas" Code Example (Clean & Neat)

Since you asked for complete and neat code, here is a simple example of **Dependency Inversion** and **Single Responsibility** working together.

```php
namespace App\Services;

interface PaymentGateway {
    public function process(int $amount): bool;
}

class StripeGateway implements PaymentGateway {
    public function process(int $amount): bool {
        // Stripe logic here
        return true;
    }
}

class OrderService {
    protected $gateway;

    public function __construct(PaymentGateway $gateway) {
        $this->gateway = $gateway;
    }

    public function createOrder(array $data) {
        // Single Responsibility: This class only handles order flow
        $this->gateway->process($data['amount']);
    }
}

```

No comments, just clean logic. By type-hinting the `PaymentGateway` interface in the constructor, we follow DIP. By moving payment logic out of the Controller, we follow SRP.
